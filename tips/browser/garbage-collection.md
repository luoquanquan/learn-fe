# 垃圾回收机制

所谓垃圾回收机制，就是浏览器回收内存的方案。在浏览器发展历程中充当了主流 GC 的方案主要有：引用计数法和标记清除法

## 引用计数法

比较初级的垃圾回收方案，目前几乎已经被完全淘汰。在引用计数法的机制下，内存中每个值都会对应一个引用计数。当垃圾回收器感受到某个值的引用计数已经为 0 时，就判定为该变量已经是无用的变量了。对应的内存就会被释放掉。

```js
// 创建一个对象 "罗小螺", 并且赋值给 man 变量, 此时对象的引用次数为 1
let man = {
  name: '罗小螺',
  age: 18
}

// 添加一个 bakUp 也指向了 "罗小螺" 对象, 此时对象的引用次数为 2
let bakUp = man

// 断开 man 对 "罗小螺" 的引用, 此时对象的引用次数为 1
man = null

// 断开 bakUp 对 "罗小螺" 的引用, 此时对象的引用次数为 0, 被 GC 认为
// 是无用对象, 垃圾收集释放内存
bakUp = null
```

如上代码，我们创建一个对象，并且赋值给 man 时，它有一个引用，所以不会被删除，再指定了 backUp 也指向它的时候。引用变成两个。只有当两个指向对象的句柄全部置为 null 的时候，GC 认为该对象没有了引用，将其删除内存释放⋯⋯

### 引用计数法的缺点

```js
// 函数 func 中定义了 圈圈 和 罗小螺 两个对象
const func = () => {
  const quanquan = { name: '圈圈' }
  const xiaoluo = { name: '罗小螺' }
}

// 执行完了 func 之后 圈圈 和 罗小螺 两个对象都会被回收
func()
```

函数作用域会执行完成后回收变量，所以 func 执行完成后quanquan 和 xiaoluo引用断开对象待释放。但是如果我们稍微改动一下代码如下：

```js
// 函数 func 中定义了 圈圈 和 罗小螺 两个对象
const func = () => {
  const quanquan = { name: '圈圈' }
  const xiaoluo = { name: '罗小螺' }

  // 通过循环引用, 圈圈 和 罗小螺 互相引用彼此. 引用计数法统计对象
  // 始终保持着一个引用便无法实现回收
  xiaoluo.friend = quanquan
  quanquan.friend = xiaoluo
}

// 按理说这里执行完了 func 之后 圈圈 和 罗小螺 两个对象都会被回收
func()
```

在函数体中创建的两个对象分别添加了对彼此的引用(也就是人们常说的循环引用)。当函数执行完毕后，原本 quanquan 和 xiaoluo 变量被回收了，但是两个对象互相引用了彼此，所以始终存在引用，内存无法释放

## 标记清除法

由于引用计数法，存在上述问题，浏览器都开始转为使用标记清除法来实现垃圾回收。在标记清除法中一个变量是否还有用的判断标准就是它是否可以被触达。该算法有两个阶段：

- 标记阶段：垃圾回收器会先找到根对象，从根对象出发，扫描所有可以通过根对象触及的变量，并将这些变量标记为可抵达
- 清除阶段，没有被标记为可抵达的对象，都被认为是不需要的，统一清除并释放内存。

```js
// 函数 func 中定义了 圈圈 和 罗小螺 两个对象
const func = () => {
  const quanquan = { name: '圈圈' }
  const xiaoluo = { name: '罗小螺' }

  xiaoluo.friend = quanquan
  quanquan.friend = xiaoluo
}

// 还是这张图, 函数执行完毕后从 window 算起, 无法
// 找到 圈圈 和 罗小螺 两个对象, 所以认为是无用的对象
// 将其回收
func()
```

## 常见的内存泄露原因

1. 函数作用域内定义变量未使用声明符，导致生成了全局变量
2. 忘记清除的 setInterval 和 setTimeout
3. 移除 Dom 未删除其引用
```js
// 创建一个变量, 并引用 DOM 元素
const dom = document.getElementById('dom')

// 移除 dom 元素, 但是没有删除变量的引用
document.body.removeChild(dom)

// 小老弟, 我还在
console.log(dom)
```
