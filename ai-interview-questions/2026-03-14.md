## 1. 在 EIP-7702 中，授权签名后 EOA 的 nonce 增加了，但 delegation designator 未变，这会导致什么安全问题？如何防范？

**考点**: EIP-7702 深入 + Replay 攻击变种

**答案要点**:

**问题**: 
- 用户签名授权 `delegation designator = 0xABC`，nonce 从 5 变成 6
- 攻击者可以重放旧签名（nonce=5 的授权），重新设置 delegation，甚至取消当前的 delegation

**防范 - 版本 nonce 设计**:
```solidity
// EIP-7702 设计中，授权消息应包含：
struct Authorization {
    address chainId;
    address contractAddress;  // 要委托的合约
    uint256 nonce;            // EOA 的当前 nonce
    uint256 yParity;
    uint256 r;
    uint256 s;
}
```

- 链上验证 `ecrecover` 签名的 nonce 必须匹配当前账户 nonce
- 一旦授权被应用，nonce 增加，旧授权自动失效
- 撤销授权 = 授权给零地址 `address(0)`

## 2. Optimistic Rollup 的 7 天挑战期内，排序器（Sequencer）作恶提交无效状态根，验证者如何构造欺诈证明（Fraud Proof）？具体要提交哪些数据？

**考点**: L2 机制 / 欺诈证明

**答案要点**:

**欺诈证明流程**:

```
1. 验证者检测：L1 上的状态根与本地执行结果不符
2. 提交断言：在 L1 合约上质押并声明 "状态根 X 是错误的"
3. 单步证明（Interactive Fraud Proof）:
   - 双方通过二分查找定位到具体哪个 opcode 执行有误
   - 最终提交该 opcode 的输入状态（memory, stack, storage）和预期输出
4. L1 合约执行该 opcode 验证结果
```

**关键数据**:
- **State Root**: 争议的状态根
- **Pre-state**: 执行前的完整状态（账户余额、nonce、storage）
- **Transaction**: 导致状态变化的交易
- **Merkle Proof**: 证明特定账户/存储槽在状态树中的值
- **Execution Trace**: 逐步执行的中间状态（用于二分查找）

**现代优化（Cannon/Bedrock）**:
- 使用 MIPS/SPARC 虚拟机执行 L2 代码
- 只需提交单条指令的输入/输出，L1 模拟执行验证

## 3. 零知识证明（ZKP）在 Web3 中的应用场景有哪些？zk-SNARK 和 zk-STARK 的核心区别是什么？为什么 ZK Rollup 选择其中一种？

**考点**: 零知识证明（进阶密码学）

**答案要点**:

**应用场景**:
1. **隐私交易**: Tornado Cash（证明知道密钥，但不暴露是哪个）
2. **ZK Rollup**: 证明批量交易执行正确，L1 只需验证证明
3. **身份验证**: 证明是某 DAO 成员，不暴露具体是谁
4. **合规**: 证明交易不涉及黑名单地址，不暴露交易详情

**zk-SNARK vs zk-STARK**:

| 特性 | zk-SNARK | zk-STARK |
|------|----------|----------|
| 可信设置 | 需要（toxic waste） | 不需要 |
| 证明大小 | 小（~200 bytes） | 大（~50KB） |
| 验证时间 | 快（几毫秒） | 较慢（但仍是线性的） |
| 量子安全 | 否（基于椭圆曲线） | 是（基于哈希） |
| 复杂度 | 较高（需电路设计） | 较简单 |

**ZK Rollup 的选择**:
- **zkSync/StarkNet**: STARK（无信任假设，抗量子）
- **Polygon zkEVM/Scroll**: SNARK（证明小，L1 验证成本低）
- 权衡：STARK 更安全，SNARK 更便宜（L1 calldata 费用）

## 4. 闪电贷（Flash Loan）的实现原理是什么？为什么它只能在同一个区块内完成？请写出核心智能合约逻辑。

**考点**: DeFi 机制 / 原子性

**答案要点**:

**原理**:
闪电贷利用交易的**原子性**——如果交易回滚，所有状态变更都会被撤销。

**执行流程**:
```
1. 用户调用 FlashLoan 合约的 execute()
2. 合约检查：当前区块内是否已发放
3. 合约发送大额代币给用户
4. 调用用户的回调函数（executeOperation）
   - 用户在回调中进行套利/清算操作
5. 检查：用户是否归还本金 + 手续费
6. 如果没有归还，revert() 整个交易
```

**核心合约逻辑**:
```solidity
function flashLoan(
    IERC3156FlashBorrower receiver,
    address token,
    uint256 amount,
    bytes calldata data
) external override {
    uint256 fee = _flashFee(token, amount);
    uint256 balanceBefore = IERC20(token).balanceOf(address(this));
    
    require(balanceBefore >= amount, "Insufficient liquidity");
    
    // 发放贷款
    IERC20(token).transfer(address(receiver), amount);
    
    // 调用借款人的回调
    require(
        receiver.onFlashLoan(msg.sender, token, amount, fee, data) 
            == CALLBACK_SUCCESS,
        "Invalid flashloan callback"
    );
    
    // 验证归还
    uint256 balanceAfter = IERC20(token).balanceOf(address(this));
    require(balanceAfter >= balanceBefore + fee, "Repay failed");
}
```

**为什么必须同区块**:
- 智能合约无法跨区块追踪"谁借了款未还"
- 只有交易的即时回滚机制能保证安全
- 没有抵押品，必须原子性归还

## 5. EVM 存储布局（Storage Layout）中，动态数组和映射的存储槽如何计算？如何通过 `eth_getStorageAt` 读取映射中特定键的值？

**考点**: EVM 底层 / 存储布局

**答案要点**:

**存储槽计算规则**:

1. **基础类型**: 顺序从 slot 0 开始，打包（packing）优化
   ```solidity
   uint128 a;  // slot 0: bytes 0-15
   uint128 b;  // slot 0: bytes 16-31
   uint256 c;  // slot 1
   ```

2. **动态数组**: slot 存储长度，数据从 `keccak256(slot)` 开始
   ```solidity
   uint256[] arr;  // slot 0 存 length
   // arr[i] 存储在 keccak256(0) + i
   ```

3. **映射**: `keccak256(key . slot)`（`.` 表示拼接）
   ```solidity
   mapping(address => uint) balances;
   // balances[addr] 存储在 keccak256(addr . 1)
   ```

**通过 RPC 读取映射值**:
```javascript
const Web3 = require('web3');
const web3 = new Web3('http://localhost:8545');

const addr = '0x1234...';  // 要查询的地址
const mappingSlot = 1;      // balances 变量的 slot 位置

// 计算存储位置
const key = web3.utils.padLeft(addr, 64);
const slot = web3.utils.padLeft(web3.utils.numberToHex(mappingSlot), 64);
const concatenated = key + slot.slice(2);  // 去掉 0x 后拼接
const storageSlot = web3.utils.keccak256(concatenated, {encoding: 'hex'});

// 读取存储
const value = await web3.eth.getStorageAt(contractAddress, storageSlot);
console.log('Balance:', web3.utils.fromWei(value));
```

## 6. 代理模式（Proxy Pattern）中，Transparent Proxy 和 UUPS（Universal Upgradeable Proxy Standard）的核心区别是什么？为什么 UUPS 更节省 Gas？

**考点**: 合约升级模式

**答案要点**:

**Transparent Proxy**:
```solidity
// 代理合约检查 msg.sender
if (msg.sender == admin) {
    // 调用管理函数（升级逻辑）
} else {
    // delegatecall 到实现合约
}
```

**UUPS**:
```solidity
// 代理合约无脑 delegatecall
assembly {
    calldatacopy(0, 0, calldatasize())
    let result := delegatecall(gas(), impl, 0, calldatasize(), 0, 0)
}

// 升级逻辑写在实现合约里
function upgradeTo(address newImplementation) external {
    require(msg.sender == admin, "Not admin");
    // 更新存储槽中的实现地址
}
```

**关键区别**:

| 特性 | Transparent | UUPS |
|------|-------------|------|
| 升级逻辑位置 | 代理合约 | 实现合约 |
| Admin 检查 | 代理合约执行 | 实现合约执行 |
| Gas 成本 | 高（每次调用都检查） | 低（delegatecall 便宜）|
| 实现合约要求 | 无 | 必须继承 UUPS 升级逻辑 |
| 风险 | 较低 | 实现合约若没写升级逻辑，合约永久锁定 |

**UUPS 省 Gas 原因**:
- Transparent: 每次调用都要在代理层做 `if-else` 检查
- UUPS: 代理层只做 delegatecall，逻辑在实现层，大部分调用不需要 admin 检查

## 7. 默克尔帕特里夏树（Merkle Patricia Trie）在以太坊中有哪三棵树？交易收据的布隆过滤器（Bloom Filter）是如何构造的？它的作用是什么？

**考点**: 以太坊数据结构

**答案要点**:

**三棵树**:

1. **状态树（State Trie）**: 账户地址 → 账户状态（balance, nonce, codeHash, storageRoot）
2. **存储树（Storage Trie）**: 每个合约独立的键值对存储
3. **交易树（Transaction Trie）**: 区块内所有交易
4. **收据树（Receipts Trie）**: 交易执行结果和日志

**布隆过滤器构造**:
```go
// 伪代码
func CreateBloom(receipts Receipts) Bloom {
    var bin Bloom
    for _, receipt := range receipts {
        for _, log := range receipt.Logs {
            // 对 address 和每个 topic 计算布隆过滤器
            bin.Add(log.Address.Bytes())
            for _, topic := range log.Topics {
                bin.Add(topic.Bytes())
            }
        }
    }
    return bin
}

func (b *Bloom) Add(data []byte) {
    // 对数据进行 3 次哈希，设置对应位
    for i := 0; i < 3; i++ {
        idx := bloomBits(data, i) % 2048  // 2048 bits = 256 bytes
        b[idx/8] |= 1 << (idx % 8)        // 设置位
    }
}
```

**作用**:
- **快速过滤**: 轻节点无需下载完整收据，只需 256 bytes 的布隆过滤器
- **检查事件存在性**: 客户端可以本地判断某事件"可能在该区块中"，然后请求完整数据验证
- **假阳性可能**: 布隆过滤器可能误判（存在），但不会漏判（不存在）

## 8. 账户抽象（ERC-4337）的 EntryPoint 合约如何验证 UserOperation 的有效性？Paymaster 如何代付 Gas？

**考点**: ERC-4337

**答案要点**:

**UserOperation 结构**:
```solidity
struct UserOperation {
    address sender;          // 智能合约钱包地址
    uint256 nonce;
    bytes initCode;          // 首次部署时使用
    bytes callData;          // 实际执行的操作
    uint256 callGasLimit;
    uint256 verificationGasLimit;
    uint256 preVerificationGas;
    uint256 maxFeePerGas;
    uint256 maxPriorityFeePerGas;
    address paymaster;       // 代付者（可选）
    bytes paymasterData;     // 代付者数据
    bytes signature;
}
```

**验证流程**:
```
1. EntryPoint.simulateValidation()
   - 检查 UserOp 格式
   - 调用 sender.validateUserOp()
   - 验证签名和 nonce
   - 如果是 paymaster，调用 paymaster.validatePaymasterUserOp()

2. EntryPoint.handleOps()
   - 执行所有 UserOp
   - 如果失败，从 sender 或 paymaster 押金中扣除 gas 费用
```

**Paymaster 代付逻辑**:
```solidity
function validatePaymasterUserOp(
    UserOperation calldata userOp,
    bytes32 userOpHash,
    uint256 maxCost
) external returns (bytes memory context, uint256 validationData) {
    // 1. 检查代付条件（例如：用户持有特定 NFT）
    require(checkEligibility(userOp.sender), "Not eligible");
    
    // 2. 检查 paymaster 押金是否足够支付 maxCost
    require(paymasterDeposit >= maxCost, "Insufficient deposit");
    
    // 3. 返回上下文（用于后续 postOp）
    return (abi.encode(userOp.sender), 0);
}

function postOp(
    PostOpMode mode,
    bytes calldata context,
    uint256 actualGasCost
) external {
    // 实际扣除用户代币或记录债务
    address user = abi.decode(context, (address));
    deductToken(user, actualGasCost);
}
```

## 9. 预编译合约（Precompiled Contracts）在 EVM 中的地址范围是什么？ecrecover 和 sha256 预编译的实现原理是什么？为什么它们比普通合约执行更快？

**考点**: EVM 预编译

**答案要点**:

**地址范围**: `0x01` 到 `0x0a`（共 9 个）

| 地址 | 功能 |
|------|------|
| 0x01 | ecrecover (ECDSA 恢复) |
| 0x02 | SHA2-256 |
| 0x03 | RIPEMD-160 |
| 0x04 | identity (数据复制) |
| 0x05 | modexp (大数模幂) |
| 0x06 | ecAdd (椭圆曲线点加) |
| 0x07 | ecMul (椭圆曲线点乘) |
| 0x08 | ecPairing (配对检查) |
| 0x09 | blake2f |

**为什么更快**:

1. **原生代码执行**: 
   - 普通合约: EVM 字节码逐条解释执行
   - 预编译: Go/C++ 原生实现，直接调用

2. **固定 Gas 成本**:
   ```go
   // go-ethereum/core/vm/contracts.go
   func (c *ecrecover) Run(input []byte) ([]byte, error) {
       // 直接调用 secp256k1.RecoverPubkey
       // 无需 EVM 栈操作
   }
   ```

3. **免存储开销**: 无代码存储、无状态树查找

**ecrecover 实现**:
```go
func (c *ecrecover) Run(input []byte) ([]byte, error) {
    const ecRecoverInputLength = 128
    input = common.RightPadBytes(input, ecRecoverInputLength)
    
    // 提取 hash, v, r, s
    r := new(big.Int).SetBytes(input[64:96])
    s := new(big.Int).SetBytes(input[96:128])
    v := input[63] - 27  // 转换 27/28 到 0/1
    
    // 使用 btcec 库恢复公钥
    pk, err := crypto.Ecrecover(input[:32], append(input[64:128], v))
    if err != nil {
        return nil, nil
    }
    
    // 返回地址（公钥哈希后20字节）
    return common.LeftPadBytes(crypto.Keccak256(pk[1:])[12:], 32), nil
}
```

## 10. 在构建跨链桥（Bridge）时，如何保证"发送链燃烧/锁定资产 → 接收链铸造/释放资产"的原子性？如果接收链交易失败，有什么回滚机制？

**考点**: 跨链桥设计

**答案要点**:

**原子性保证 - 哈希时间锁合约（HTLC）**:
```solidity
// 发送链
function lock(
    bytes32 hash,
    address recipient,
    uint256 timeout
) external payable {
    // 资金锁定在此合约
    locks[hash] = Lock(msg.value, recipient, block.number + timeout);
}

// 接收链  
function mint(
    bytes32 preimage,
    address recipient
) external {
    require(sha256(preimage) == knownHash, "Invalid preimage");
    // 铸造代币给 recipient
}
```

**现代桥设计（信任假设）**:

| 类型 | 机制 | 信任假设 |
|------|------|----------|
| 轻客户端桥 | 在链 A 验证链 B 的区块头 | 只需要信任源链共识 |
| 多签桥 | 验证者委员会多签 | 信任验证者集合 |
| ZK 桥 | ZK 证明源链状态有效性 | 密码学安全，最低信任 |

**失败处理机制**:

```
1. Optimistic 验证（如 Nomad）:
   - 消息提交后等待 30 分钟挑战期
   - 如果失败，发送链可发起 fraud proof 回滚

2. 两阶段提交:
   - 阶段1: 在接收链"预执行"，检查是否成功
   - 阶段2: 确认后才在发送链最终化

3. 资金保险:
   - 接收链失败 → 资金留在发送链合约
   - 用户可通过超时赎回（HTLC 模式）
```
